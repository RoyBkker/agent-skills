# Fix Merge Conflicts

## Scope

This skill auto-resolves **mechanical** conflicts only. It never guesses at conflicting business logic — those are always explained and proposed to the user, never silently applied. An unresolved conflict is safe; a wrongly-resolved one is not.

## Quick start

1. Confirm merge/rebase state: `git status`
2. List conflicted files: `git diff --name-only --diff-filter=U`
3. Classify each file (see Workflow)
4. **Explain every conflict and the planned resolution before touching any file**
5. Apply the safe resolutions; leave logic conflicts for the user to approve
6. Run the project's checks (typecheck/tests/format) to verify the resolution
7. Show what changed and wait for confirmation before staging/committing

## Workflow

Copy this checklist and check off items as you progress:

```
Conflict Resolution Progress:
- [ ] Step 1: Establish context (merge or rebase? conflict style?)
- [ ] Step 2: Classify every conflicted file
- [ ] Step 3: Explain every conflict + proposed resolution — before editing anything
- [ ] Step 4: Apply only the approved resolutions
- [ ] Step 5: Run the project's checks to verify the resolution
- [ ] Step 6: Summarize changes, confirm before staging/committing
```

### 1. Establish context
- `git status` — determine if mid-merge or mid-rebase (affects continue/abort commands, and which side is "ours" vs "theirs" — these swap during a rebase).
- If `merge.conflictStyle` isn't `zdiff3`, note it: diff3-style markers show the common ancestor, making both classification and explanation far more reliable. Suggest `git config merge.conflictStyle zdiff3` for this repo if unset — don't change it without asking.
- `git diff --name-only --diff-filter=U` for the full conflict list.

### 2. Classify each conflicted file
- [ ] **Lockfile** (package-lock.json, yarn.lock, pnpm-lock.yaml, Cargo.lock, Gemfile.lock, poetry.lock, go.sum, composer.lock) → regenerate, don't hand-merge. See REFERENCE.md.
- [ ] **Generated/build output** (dist/, snapshot files, autogenerated code) → regenerate from source, or take the canonical side if regeneration isn't available. Flag if it's unclear which side is canonical.
- [ ] **Whitespace/formatting-only** — both hunks resolve to equivalent code after formatting → merge, then re-run the formatter.
- [ ] **Non-overlapping additions** — both sides added different, unrelated lines that git grouped into one hunk only because they're adjacent → keep both.
- [ ] **Logic conflict** — same line(s)/function modified with different intent → never auto-apply. Read `git log -p` for both sides to understand intent before proposing a resolution.

### 3. Explain before acting
For **every** conflicted file — safe or not — summarize in plain language before making any edit:
- What each side actually changed, and why (infer intent from the commit messages/diff context, not just the raw hunk).
- The proposed resolution: for safe files, state the mechanical fix about to be applied; for logic conflicts, propose a specific resolution (e.g., "combine both: keep the retry logic from main and the new timeout param from this branch") rather than just saying "needs manual review."

Wait for the user to greenlight the plan before editing anything, especially if any classification is uncertain.

**Example — mechanical conflict:**

> `package-lock.json`: both branches added dependencies, git can't merge the lock hashes. Proposed: regenerate from `package.json` via `npm install` (see REFERENCE.md).

**Example — logic conflict:**

> `src/payments/retry.ts`: main added exponential backoff to `sendPayment`; this branch added a `timeoutMs` parameter to the same function. Proposed: keep main's backoff loop and thread `timeoutMs` through it as the per-attempt timeout. Apply this?

### 4. Apply resolutions
- Safe files: edit out the conflict markers per the explained plan. Don't `git add` yet.
- Logic conflicts: only apply the proposed resolution after the user explicitly approves it — either the plan from step 3 or their own correction.

### 5. Verify
Run the project's own checks against the resolved state — typically typecheck, then tests, then format — before proposing to finalize. A resolution can look correct and still break the build; this catches that before it's committed. Fix anything the resolution broke, then re-run.

### 6. Summarize and confirm
Before staging or committing anything, show:
- Files resolved, and how (one line each, matching what was proposed in step 3).
- Verification results from step 5.
- Any files still open, with a pointer to the remaining conflicting hunks.
Wait for explicit confirmation before running `git add` + `git commit` (merge) or `git rebase --continue`.

### 7. Never do these without being asked
- Don't use `git checkout --ours/--theirs` on a whole file unless you've already verified the file is 100% mechanical — it silently discards the other side even for logic files.
- Don't `git merge --abort` / `git rebase --abort` to "start over" without confirming — it discards any resolution work already done.
- Don't force-push after rebase resolution without confirming with the user first.
